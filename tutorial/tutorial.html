FX Bug Toolkit

Investigate Firefox bugs, fast. Point it at a bug and it pulls the report, searches Gecko, reads profiles and logs, checks the spec, and writes up a clean, link-rich investigation — then lets you browse them all.

GitHub repo ↗ Get started ↓

What it does

One hub command — /bug-start — drives an end-to-end investigation. Everything else supports it.

  • Pulls the Bugzilla report and searches the Gecko codebase for you.
  • Analyzes Firefox Profiler captures and log files (deep media/EME/CDM know-how).
  • Checks spec conformance and writes a structured, searchfox-linked investigation file.
  • A local web viewer to search and re-read every past investigation.
  • Runs your weekly A/V bug triage — drafts the comments, sets priority/severity, and gives you a dashboard to review and apply them.
It investigates and diagnoses — it doesn't write or land patches. Works for any Firefox component; the bundled know-how runs deepest for A/V / media. Runs on Windows, macOS, and Linux.

Quick start

1

Install the plugin

No clone needed — add it straight from GitHub:

claude plugin marketplace add alastor0325/fx-bug-toolkit
claude plugin install fx-bug-toolkit@fx-bug-toolkit

Prefer a local copy to read or hack on? git clone the repo first, then claude plugin marketplace add ./fx-bug-toolkit.

2

Restart Claude Code

Plugins load at startup. After restarting, type / and you'll see the toolkit's commands (each shows fx-bug-toolkit as its source).

3

Run setup

/init

It checks your environment and offers to install anything missing (it can install the core CLIs for you). When it says “Core investigation ready,” you're good to go.

Investigate a bug

The one command you'll use most — give it a Bugzilla bug number:

/bug-start 2044320

It investigates the bug end to end, writes an investigation file, and finishes by opening the viewer with a deep link straight to that bug:

View this investigation → http://127.0.0.1:8777/viewer.html#2044320

You can also run the analyzers on their own:

/analyze-profile https://share.firefox.dev/public/...   # a Profiler capture
/check-log /path/to/firefox.log                      # a Firefox log

To revise an existing write-up, just ask in plain language — “update the investigation for bug 2044320 with this new root cause.”

Browse investigations

/browse

Builds an index of every investigation (across all your folders) and serves a fast, searchable viewer on 127.0.0.1 — your data never leaves the machine.

Everything on screen is genuine output from /bug-start on a real, public bug — frontmatter summary, root-cause callout, searchfox-linked affected files, and the rendered analysis.
Your data, offline. Every investigation is saved as plain Markdown on your own machine — nothing is uploaded, so you have full control (read, edit, move, version them). The location is configurable via FX_BUG_INVESTIGATION_DIR. Optionally connect a shared team knowledge base via the firefox-wiki-plugin — purely additive; it works fully offline without it.

Weekly triage

A second workflow, for the Audio/Video bug-triage rotation: /triage does the analysis and drafts every action; /triage-dashboard is where you review and apply them.

Draft the triage

/triage                    # full session: watched bugs + new bugs from the last 14 days
/triage 2044320            # just one bug
/triage https://bugzilla.mozilla.org/buglist.cgi?...   # every bug in a query

For each in-scope bug it checks information completeness, looks for duplicates, assesses priority & severity, drafts a needinfo question when something's missing, and wires up meta-bug blocking. It writes each decision as a pending draft — it never touches Bugzilla on its own. You review and apply from the dashboard.

Scoped to the Audio/Video components by default; the set is configurable on first run. Like the rest of the toolkit, all I/O goes through local CLIs and your data stays on your machine.

Review & apply on the dashboard

/triage-dashboard

Opens a local web UI over the drafts on 127.0.0.1:8765. Each bug becomes a card: the current Bugzilla state, the proposed comment, the priority/severity it'll set, and — when you've also run /bug-start on it — the investigation findings inline. Tabs split the work into Analyzed, Needs Info, Close / Reassign, and Awaiting reply.

Not happy with a draft? Type feedback into the card's box to have it re-drafted, toggle whether to CC/needinfo yourself, then hit Apply — the dashboard queues a bugzilla-cli apply that you confirm in the terminal. Nothing is posted to Bugzilla until you say yes.

The dashboard is optional and installs itself the first time you run /triage-dashboard (a small FastAPI app in a managed virtualenv) — /triage works without it, it just writes the drafts to disk.

A few helpers (spec-check, download-guard, source-links, the gecko-navigator agent) run behind the scenes — Claude calls them from the commands above; you don't invoke them directly.

Optional: the shared wiki

Install the separate firefox-wiki-plugin and clone its content to ~/firefox-wiki for faster starts and compounding team knowledge — Claude checks known component behavior before reading code, and contributes new findings back. The toolkit detects it automatically and works perfectly fine without it.
